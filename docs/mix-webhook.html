Best Practices Ltd · JMG Webhook Integration

Webhook connection

Best Practices Ltd provides the endpoint below as the only authorised channel for MiX to deliver push data to the JMG fleet integration. Configure MiX Integrate to POST JSON here whenever events occur or data changes — see page 2 for what to send and when.

Storage (current setup)

Webhook payloads are saved to webhook-data/ on the server — regular files, no database cost. This is separate from polling output (public/, etc.). The MiX URL does not change if storage is upgraded later.

PhaseStorageStatus
1 — Demo (now)webhook-data/ filesActive — default, no extra cost
2 — Live dashboardRead from webhook storePlanned — same URL, wire UI to webhook data
3 — PostgreSQLManaged databaseFuture — only if you add a paid DB later; same MiX URL

No DATABASE_URL is required for the demo. Leave WEBHOOK_STORE=file (or unset).

Authorised delivery endpoint

POST https://jmg.bestpracticesltd.com.ng/api/mix-webhook

All push data for JMG must be delivered here via HTTPS POST with a JSON body (see page 2). Traffic to any other URL will not be processed.

Health check

GET https://jmg.bestpracticesltd.com.ng/api/mix-webhook/health

Optional. Unauthenticated status check to confirm the endpoint is reachable before enabling push. This URL does not accept event data.

Authentication

Every POST to the authorised endpoint must include the shared secret issued during onboarding:

HeaderValue
Content-Typeapplication/json
x-webhook-secretProvided separately during onboarding

Requests without a valid secret receive 401 Unauthorized and are discarded.

Data requirements

Configure MiX Integrate to push JSON to our webhook (page 1) whenever relevant data is created or updated for the JMG organisation. Your integration work is to define which events trigger delivery and ensure the payloads below are POSTed to the authorised endpoint. We do not pull webhook data — delivery is push-only from MiX.

How push delivery works

MiX Integrate sends an HTTPS POST to our endpoint each time something happens in the fleet. There are three delivery patterns:

PatternWhen MiX should POSTEntity types
Real-time events Immediately when the event occurs for a JMG asset Events (panic, harsh braking, overspeeding, etc.)
Lifecycle updates When a trip completes, or when master data is created or changed Trips, drivers, vehicles, sites
Live position While moving: POST a position every 5 minutes. When stopped or idle: every 30 minutes. Position

Organisation scope

Only data for the following organisation may be sent to this integration. Do not route events from other organisations to these endpoints.

The organisation ID will be supplied in a separate secure channel together with the webhook secret.

Payload structure

Each POST body is JSON. Send a single entity object, an array of entities, or a wrapper with an events, trips, or data array — whichever matches your Integrate push configuration. Field names and nesting should follow the MiX Integrate API entity definitions.

Supported entity types and how we identify them:

EventEventTypeIdEventId, AssetId, Position

VehicleAssetId + registration/descriptionRegistrationNumber, Description
DriverDriverId + name/phoneName, MobileNumber
PositionAssetId + coordinatesLatitude, Longitude, Timestamp
SiteGroupId + TypeName, zone fields

Events — configure and push

Events are the primary integration surface. In MiX Integrate, enable webhook push for each EventTypeId listed below for the JMG organisation. When any of these events fire — for example, a driver triggers a panic button or a vehicle exceeds the speed limit — POST the event entity to our endpoint without delay. Do not batch events for later delivery unless your platform requires it; near-real-time delivery is expected.

Each event object must include at least:

FieldTypeDescription
EventIdstringUnique event identifier
EventTypeIdstringMiX event type identifier
AssetIdstringVehicle / asset identifier
EventDateTimeISO 8601When the event occurred

Recommended where available: DriverId, ReceivedDateTime, Position (Latitude, Longitude, FormattedAddress).

Trips — push on completion

When a driver completes a trip for a JMG asset, POST the trip entity. Required identifiers: TripId, AssetId, TripStart, TripEnd, and DistanceKilometers. Include DriverId where available.

Reference data — drivers, vehicles, sites

Master data should be pushed so our dashboard can resolve IDs to names and registrations:

  • Initial sync: Before go-live, push the current driver, vehicle, and site records for the JMG fleet.
  • Ongoing updates: POST whenever a driver is added or their details change, a vehicle is registered or updated, or a site/geofence is created or modified.

Drivers are identified by DriverId with Name and/or MobileNumber. Vehicles by AssetId with RegistrationNumber and/or Description. Sites by GroupId and Type.
